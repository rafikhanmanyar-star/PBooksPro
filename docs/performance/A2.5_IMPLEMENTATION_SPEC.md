# Task PERF-A2.5.1 — WhatsAppContext Memoization

## 1. Executive Summary

`WhatsAppProvider` creates a **new context value object on every render** despite stable `useCallback` handlers. With **~38 consumer files**, each provider rerender (e.g. when `isOpen` toggles or `useAuth` parent updates) forces React to treat the context value as changed. Memoizing the provider value eliminates spurious context propagation while preserving identical public API.

## 2. Current State

### Provider architecture

**File:** `context/WhatsAppContext.tsx`

```
WhatsAppProvider
├── useAuth() → isAuthenticated (subscription to AuthContext)
├── useState: isOpen, contact, phoneNumber, initialMessage
├── useCallback: openChat, closeChat (stable)
├── useEffect: close on logout
├── Provider value={{ openChat, closeChat, isOpen }}  ← NEW OBJECT EVERY RENDER
├── children
└── WhatsAppSidePanel (always mounted as sibling)
```

### Provider value creation

```tsx
// Line 52 — inline object literal, no useMemo
<WhatsAppContext.Provider value={{ openChat, closeChat, isOpen }}>
```

### Consumer count

| Metric | Count |
|--------|------:|
| Files importing `useWhatsApp()` | **38** |
| Typical usage | Call `openChat()` only (~90% of consumers) |
| Consumers reading `isOpen` | Provider + `WhatsAppSidePanel` only |

**Consumer categories:**

- Layout: `Header.tsx`
- Invoices/bills: `InvoiceBillItem`, `BillsPage`, payment modals
- Ledgers/payouts: `OwnerLedger`, `BrokerLedger`, payout modals
- Reports: `ClientLedgerReport`, `TenantLedgerReport`, `OwnerPayoutsReport`, etc.
- Procurement, payroll, rental, project modules

### Current rerender behavior

| Trigger | Provider rerenders | Consumer impact |
|---------|-------------------|-----------------|
| `openChat()` / `closeChat()` | Yes (`isOpen`, contact state) | **Expected** — panel opens/closes |
| Auth `isAuthenticated` change | Yes (effect + possible parent) | All 38 consumers receive new value reference |
| Parent tree rerender (unrelated) | If provider parent rerenders | New value object → all consumers re-evaluate context |
| Consumer calls only `openChat` | N/A | Still subscribed to full context; unstable value prevents bailout |

**Note:** Most consumers destructure `{ openChat }` only. React Context compares value by reference (`Object.is`). Unstable wrapper object defeats memoization of child components that depend on context.

## 3. Proposed Change

### Proposed architecture

```
WhatsAppProvider
├── (unchanged state + callbacks)
├── contextValue = useMemo(
│     () => ({ openChat, closeChat, isOpen }),
│     [openChat, closeChat, isOpen]
│   )
└── Provider value={contextValue}
```

**Optional follow-up (out of A2.5.1 scope — document only):**

- Split into `WhatsAppActionsContext` (stable forever) + `WhatsAppPanelContext` (`isOpen` only) — defer to A2.4.

### Implementation notes

- Add `useMemo` import from React.
- Do **not** include `contact`, `phoneNumber`, `initialMessage` in context value (already passed directly to `WhatsAppSidePanel`).
- No changes to `openChat` / `closeChat` signatures or `WhatsAppSidePanel` props.

## 4. Files Affected

| File | Change type |
|------|-------------|
| `context/WhatsAppContext.tsx` | Add `useMemo` for provider value |

**No consumer file changes required.**

## 5. Dependencies

|------------|-------|

| Blocks | Nothing |
| Blocked by | Nothing |

## 6. Risk Assessment

| Dimension | Level | Rationale |

| Functional regression | **Low** | Pure memoization; same object shape |
| Sync impact | **None** | No socket/state/sync code touched |
| Consumer breakage | **None** | API unchanged |
| Performance regression | **Negligible** | One `useMemo` per provider render |

# Task PERF-A2.5.2 — Stable Selector References

## 1. Executive Summary

`useStateSelector` wraps `useSyncExternalStore` with `getSnapshot = useCallback(() => selector(_getAppState()), [selector])`. **Inline arrow selectors** `(s) => s.transactions` allocate a new function every render, invalidating `useCallback` and forcing snapshot re-reads on every parent render. Exporting **module-level stable selector functions** for hot paths reduces CPU overhead on the most subscribed slices without changing subscription semantics.

## 2. Current State

### Hook implementation

**File:** `hooks/useSelectiveState.ts` (lines 35–37)

```typescript
export function useStateSelector<T>(selector: (state: AppState) => T): T {
    const getSnapshot = useCallback(() => selector(_getAppState()), [selector]);
    return useSyncExternalStore(_subscribeAppState, getSnapshot);
}
```

### Inline selector prevalence

| Pattern | Approx. call sites (excl. `useSelectiveState.ts` internals) |
|---------|--------------------------------------------------------------|
| Inline `(s) => s.field` or `(s) => s.field` | **~130+** |
| Already stable via named hooks (`useTransactions()`, etc.) | **~140+ files** use slice hooks |
| **Full-state selector** `useStateSelector(s => s)` | **7 files** (anti-pattern — separate concern) |

### Behavioral impact today

- **Rerender trigger:** Still correct — only when selected slice reference changes (`Object.is` on snapshot).
- **Extra cost:** On every parent render, unstable `selector` → new `getSnapshot` → `useSyncExternalStore` rebinds snapshot reader → `_getAppState()` invoked again for that hook instance.
- **Amplification:** Components with 10–14 inline selectors (Header, GlobalSearchBar, InvoicesPage) multiply this cost.
