# Task PERF-A2.1.1 — ContactsPage Virtualization

## 1. Executive Summary

`ContactsPage` renders a full HTML `<table>` with client-side sort, tab filters, tree sidebar, and a load-more cap (`displayLimit=50`, +200 per click). At 200–500+ contacts this creates 1,400–2,000+ DOM nodes and scroll jank. Virtual scroll replaces load-more entirely.

## 2. Current State

**File:** `components/contacts/ContactsPage.tsx`


| Aspect        | Detail                                                                     |
| ------------- | -------------------------------------------------------------------------- |
| Rendering     | `contacts.slice(0, displayLimit).map(...)` — inline `<table>`              |
| Sort          | 6 columns, client-side `sortConfig`                                        |
| Filters       | Text search, tab (`All`/types), tree sidebar selection                     |
| Subscriptions | 5+ inline `useStateSelector` (consider stable hooks from A2.5.2 follow-up) |
| Row actions   | Edit, ledger modal, WhatsApp, balance display                              |
| Interim cap   | `displayLimit` initial **50**, load-more **+200** (PERF-A2.5.5)            |


## 3. Proposed Change

### Option A (recommended) — SmartTable

1. Extract `ContactsTable.tsx` (or inline refactor) using `SmartTable<ContactRow>`.
2. Column defs mirroring current columns (~7); `render` column for action buttons.
3. Props: `virtualize={true}`, `virtualizeThreshold={50}`, `rowHeight={44}`, `tableHeight` from container `ref` or flex parent.
4. Parent keeps: filter/sort `useMemo` chains unchanged; pass sorted `contacts` array to table.
5. **Remove** `displayLimit` state and load-more footer — virtual scroll replaces it.
6. Stable `getRowId={(row) => row.id}`.

### Option B — Dedicated virtual table

Copy `OwnerLedger` `List` + memo row pattern if SmartTable column renderers cannot preserve row UX (e.g. complex balance cells).

## 4. Files Affected


| File                                    | Action                                                  |
| --------------------------------------- | ------------------------------------------------------- |
| `components/contacts/ContactsPage.tsx`  | Replace table body with virtual table; remove load-more |
| `components/contacts/ContactsTable.tsx` | **Create** (if Option A extracted)                      |


## 5. Risk Assessment


| Dimension             | Level      | Rationale                               |
| --------------------- | ---------- | --------------------------------------- |
| Functional regression | **Medium** | Row click, WhatsApp, ledger modal, sort |
| Sync impact           | **None**   | Read-only render path                   |
| UX change             | **Low**    | Users scroll instead of "Load more"     |


## 6. Test Cases


| ID          | Scenario                     | Expected                            |

| T-A2.1.1-03 | Tab + tree filter            | Filtered set virtualized            |
| T-A2.1.1-04 | Row click → edit modal       | Opens correct contact               |
| T-A2.1.1-05 | Socket contact create/update | Row appears/updates without F5      |


## 7. Success Criteria

- [ ] No `displayLimit` / load-more on ContactsPage.
- [ ] ≤30 table row DOM nodes at any scroll position with 500+ contacts.
- [ ] Scroll FPS ≥55 on mid hardware (Profiler spot-check).
- [ ] Build passes; no sync regression.

# Task PERF-A2.1.2 — BillsPage Virtualization

## 1. Executive Summary

`BillsPage` is the highest-complexity A2.1 target: inline `<table>`, heterogeneous row types (bills, payments, batches), bulk payment selection, tree sidebar, and 9-column sort. Interim `displayLimit=50` (A2.5.5) reduces first paint but does not fix scroll at scale.

## 2. Current State

**File:** `components/bills/BillsPage.tsx`


| Aspect        | Detail                                                                              |
| ------------- | ----------------------------------------------------------------------------------- |
| Rendering     | `filteredRows.slice(0, displayLimit).map(...)`                                      |
| Row types     | Bill rows, payment rows, batch rows — different cell layouts                        |
| Features      | Bulk pay selection (`selectedBillIds`), tree filter, date range, export/print hooks |
| Subscriptions | Full-state `useStateSelector(s => s)` — **do not widen in A2.1**                    |
| Interim cap   | Initial **50**, load-more **+200**                                                  |


## 3. Proposed Change

1. Extract `BillsTable.tsx` with SmartTable or custom `VirtualizedBillsTable`.
2. Pre-compute `filteredRows` in parent (unchanged logic).
3. **Flatten heterogeneous rows** into a unified virtual list (preferred over variable row height):
  - Each bill, payment, and batch becomes one virtual row with a `rowKind` discriminator.
  - Indentation via `depth` prop (same pattern as `VirtualizedLedgerTable`).
4. Stable `getRowId` per row kind (bill id, payment id, batch key).
5. Bulk selection: checkbox column preserved; selection state in parent unchanged.
6. Remove `displayLimit` and load-more footer.

### Fallback

If flattening breaks bulk-pay UX, evaluate `VirtualizedLedgerTable` group/child pattern with fixed row height and truncated particulars.

## 4. Files Affected


| File                              | Action                               |
| --------------------------------- | ------------------------------------ |
| `components/bills/BillsPage.tsx`  | Replace table body; remove load-more |
| `components/bills/BillsTable.tsx` | **Create**                           |


## 5. Risk Assessment


| Dimension             | Level     | Rationale                               |
| --------------------- | --------- | --------------------------------------- |
| Functional regression | **High**  | Payment batches, bulk pay, print/export |
| Sync impact           | **None**  | Render-only                             |

## 6. Test Cases

## 7. Success Criteria

- [ ] Flattened virtual rows render all row kinds correctly.

## Virtualization Design Standard

All new virtualized tables must:

- Use react-window

- Preserve search behavior

- Preserve sorting behavior

- Preserve filtering behavior

- Preserve row actions

- Preserve keyboard navigation

- Preserve synchronization updates

- Preserve React Query refreshes

No feature behavior may change.

Virtualization is a rendering optimization only.
